3 ОПИС ВИКОНАНОЇ РОБОТИ
     3.1Опис роботи:
     3.1.1 Загальні відомості
	Система обрана для виконання це Discord. Discord - це платформа для голосового, відео та текстового спілкування, що була заснована у 2015 році Джейсоном Цитроном і Станіславом Вишневським як інструмент для геймерів. Від початку сервіс задумувався як зручна альтернатива складним у налаштуванні голосовим чатам того часу: він мав дозволити гравцям спілкуватися під час ігрових сесій без необхідності виходити з гри чи перемикати вікна. Проте досить швидко платформа вийшла далеко за межі ігрової аудиторії. Щодня на серверах Discord відбувається понад 4 мільярди хвилин розмов. При цьому близько 54% активних користувачів вже не є геймерами - платформа перетворилася на універсальний простір для навчальних спільнот, творчих проєктів.
Сьогодні Discord є однією з найбільших комунікаційних платформ у світі. 
	3.1.2 Загальна характеристика, гібридна архітектура
  Discord не підпадає під жоден єдиний архітектурний стиль і це свідомо прийняте інженерне рішення. Система поєднує одразу декілька підходів, кожен із яких відповідає за свою зону відповідальності:
• Монолітний API - для стабільної бізнес-логіки 
• Мікросервісна архітектура - для real-time та масштабованих підсистем 
• Actor Model - для управління паралелізмом та відмовостійкістю 
• Event-Driven Architecture - для асинхронної взаємодії між сервісами 
• Клієнт-серверна модель - як базова топологія мережевої взаємодії
   Центральним елементом бізнес-логіки є монолітний REST API, написаний на Python. Він відповідає за автентифікацію, управління серверами, каналами, правами доступу, ботами та іншою «повільною» логікою, що не потребує мілісекундних затримок. Монолітний API обробляє основну бізнес-логіку та API-функціональність. Ця архітектурна дуальність відображає прагматичний підхід: моноліт забезпечує стабільність для усталених функцій, тоді як мікросервіси вводяться там, де потрібне незалежне масштабування або екстремальна продуктивність.
   Real-time частина системи реалізована у вигляді 20 окремих мікросервісів на Elixir та BEAM. Elixir від початку обрали для WebSocket-шлюзу, який відповідає за ретрансляцію повідомлень і real-time реплікацію. Серед цих сервісів: 
• Gateway - керування WebSocket-з'єднаннями мільйонів клієнтів
• Presence - відстеження статусу користувачів у реальному часі 
• Voice signaling - сигналізація для голосових сесій
• Push notifications - сповіщення про згадки та повідомлення
   Мікросервісний підхід дозволяє масштабувати кожен компонент незалежно залежно від навантаження.
   Фундаментом real-time системи є Actor Model, реалізований через Erlang VM. Кожен об'єкт системи - сесія користувача, guild, голосовий канал - існує як окремий легкий процес-актор.
   Користувачі підключаються через WebSocket і запускають сесійний процес, який спілкується з вузлами, що містять guild-процеси. Коли щось публікується в guild, воно розсилається кожній підключеній сесії.
   Ключові властивості BEAM, які використовує Discord:
• Ізольовані процеси: збій одного актора не поширюється на інші
• Supervision Trees: якщо процес падає, супервайзер автоматично його перезапускає
• Hot Code Reloading: оновлення коду без перезапуску сервісу та без простою
• Легкі процеси: система може утримувати мільйони одночасних акторів у пам'яті
   Взаємодія між сервісами побудована на подієво-орієнтованій архітектурі. Бекенд-сервіси Discord спілкуються асинхронно через черги повідомлень, event-driven архітектуру та pub/sub-системи. Асинхронна комунікація дозволяє сервісам бути незалежними, що покращує стійкість, масштабованість і відмовостійкість. 
   Pub та Sub у Discord реалізований через сам BEAM: guild-процес є «видавцем» подій, а сесійні процеси - «підписниками». Коли користувач надсилає повідомлення, guild-актор публікує подію і вона усім підключеним учасникам сервера.
   Між монолітним API та базою даних Discord розмістив окремий шар Data Services, написаний на Rust. Ці сервіси виступають посередниками між монолітним API та кластерами ScyllaDB. Вони відповідають виключно за доступ до даних і координацію, без жодної бізнес-логіки. 
   Rust обраний саме тут через:
• Fearless Concurrency - безпечне керування паралельними запитами без перегонів даних
• Мінімальні затримки - відсутність garbage collector, передбачуваний час відповіді
• Request Coalescing - якщо сотні користувачів одночасно запитують одне повідомлення, Data Service робить лише один запит до ScyllaDB
     Голосовий і відеозв'язок реалізований через WebRTC, але Discord суттєво модифікував стандартну бібліотеку. Для мобільних і десктопних застосунків Discord використовує єдиний C++ медіа-рушій, побудований поверх нативної бібліотеки WebRTC - спеціально адаптований під потреби користувачів. 
Оскільки кожен клієнт підключається до медіа-relay-сервера Discord, а не напряму до інших учасників, необхідність у ICE відпадає. Це забезпечує надійніше з'єднання за NAT і приховує IP-адреси учасників.
Аудіокодек: Opus, це відкритий стандарт з затримкою від 5 до 26,5 мс, обов'язковий компонент WebRTC.
Кожен об'єкт у Discord - користувач, повідомлення, сервер, канал - отримує унікальний 64-бітний Snowflake ID. Snowflake ID - це 64-бітне ціле число без знаку, яке кодує мітку часу та метадані, гарантуючи унікальність у всій системі Discord. Це дозволяє генерувати унікальні ID розподілено, без центрального координатора, і водночас сортувати об'єкти хронологічно що критично для пагінації повідомлень.
Статичний контент роздається через CDN. Discord використовує CDN для доставки статичного контенту - зображень, емодзі та вкладених файлів користувача. CDN знижує затримку й покращує швидкість доставки завдяки кешуванню контенту на edge-серверах поблизу користувачів.
     3.1.2 Основні компоненти
     Клієнти
     Discord підтримує чотири платформи з єдиним кодом бізнес-логіки: настільний, мобільні, веб-клієнт та вбудований у браузер режим. Всі вони спілкуються з бекендом через WebSocket для real-time подій і HTTPS REST для решти запитів.
     Шлюзи
     Шлюз на Elixir та BEAM - це перший рівень, що приймає всі WebSocket-з'єднання від клієнтів. Кожен користувач підключається через WebSocket і запускає сесійний процес, який взаємодіє з вузлами, де містяться guild-процеси. Шлюз відповідає за автентифікацію, heartbeat-перевірку з'єднань і первинну маршрутизацію подій.
     API Monolith
     REST API на Python обробляє «важку» бізнес-логіку: управління серверами, каналами, правами ролей, автентифікацію OAuth2, роботу з ботами. Коли користувач надсилає повідомлення, клієнт відправляє його через HTTP до API-сервісу, який записує його в базу даних і передає далі до Elixir-стека через gRPC.
     Elixir-сервіси
     Приблизно 20 мікросервісів на Elixir та BEAM реалізують усю real-time логіку. Ключові з них:
• Guild process - центральний маршрутизатор кожного Discord-сервера. Кожен онлайн-користувач має окремий Elixir «сесійний процес». Коли guild-процес отримує нове повідомлення або подію, він розсилає цю інформацію відповідним сесійним процесам. 
• Presence - відстежує статус у реальному часі.
• Relay layer - проміжний шар між guild і session для розподілу навантаження fan-out. Discord досяг підтримки понад мільйон онлайн-користувачів в одному сервері завдяки серії оптимізацій: вимкнення сповіщень для пасивних сесій знизило вартість fan-out на 90%, а новий шар relay-процесів звільнив guild-процес для обробки більшої кількості користувачів.
  Data Services
  Data Services сидять між API та кластерами ScyllaDB. Вони містять приблизно один gRPC-endpoint на кожен запит до бази даних і свідомо не містять жодної бізнес-логіки. Головна функція - request coalescing: якщо сотні користувачів одночасно запитують одні й ті самі дані, Data Service робить лише один запит до БД. Це захищає ScyllaDB від «гарячих партицій».
     Кешування
     Redis використовується для зберігання сесій, черг push-сповіщень і короткочасного кешування presence-даних. 
     Cloudflare 
     Містить весь статичний контент - аватари, емодзі, прикріплені файли з edge-серверів поблизу користувача.
     Голосова підсистема
     Окремий шар Voice relay servers приймає WebRTC-з'єднання і забезпечує голосовий/відео-зв'язок. Elixir-сервіс сигналізації керує встановленням з'єднань і розподілом між серверами. З 2024 року наскрізне шифрування DAVE E2EE.
     3.1.3 Взаємодія підсистем
     Приклад взаємодії підсистем можна побачити на діаграмі див. рис. Б.1
     Користувач натискає «відправити». Клієнт надсилає HTTP POST-запит до Gateway через постійне WebSocket-з'єднання. Gateway автентифікує запит і передає його далі через gRPC.
     Коли користувач надсилає повідомлення, клієнт відправляє його через HTTP до API-сервісу, який записує його в базу даних і передає до Elixir-стека через gRPC. На цьому етапі API виконує валідацію прав доступу, перевірку каналу, а також призначає повідомленню Snowflake ID.
     API одночасно запускає два незалежні процеси. Перший - надсилає дані до Data Services для запису в ScyllaDB. Другий - через gRPC передає подію до Guild process для розсилки. Трасування показує, що guild-сервіс розсилає повідомлення користувача асинхронно відносно обробки в API тобто збереження і доставка відбуваються паралельно.
     Guild-процес відстежує сесії всіх користувачів, що є учасниками сервера. При кожному оновленні він розсилає повідомлення всім підключеним клієнтським процесам. Для великих серверів між guild і сесіями існує додатковий relay layer, який розподіляє навантаження fan-out між кількома вузлами.
     Кожен сесійний процес отримує подію і надсилає її відповідному клієнту через WebSocket. Повідомлення з'являється на екрані отримувача.
     Окремий сценарій - це приєднання до голосовогу чату. Elixir-сервіс сигналізування встановлює WebRTC-сесію між клієнтом і найближчим Voice server. Після цього аудіо або відео потік іде напряму між клієнтом і relay, оминаючи основний API.
     3.1.4 Масштабованість та переваги архітектурних рішень
     Горизонтальне масштабування
     Одна з ключових переваг вибору Elixir та BEAM - це масштабування закладене в саму платформу. BEAM-процеси дуже легкі. Discord може запускати мільйони паралельних процесів, і кожен з них працює незалежно збій в одному не впливає на решту.
     Кожен сервіс масштабується окремо
     Elixir-сервіси спроєктовані так, щоб кожен міг масштабуватися незалежно, бо вони обробляють значно більший трафік ніж монолітний API.
     Автоматичне відновлення після збоїв
     Якщо якийсь процес падає через помилку, супервізор автоматично перезапускає його. Збій у сесії одного користувача не завалить увесь сервер.
     Компроміси
     Гібридна архітектура з трьома різними мовами дає гнучкість, але вимагає від інженерів знань одразу кількох стеків. Крім того, чим більший сервер - тим дорожче коштує розсилка повідомлень усім учасникам, і це системний виклик, який потребує постійної уваги.

4 ВИСНОВКИ
     У ході роботи було досліджено архітектуру Discord як приклад сучасної складної розподіленої системи. Визначено, що платформа використовує гібридний підхід: монолітний API, мікросервіс, та ScyllaDB як основне сховище даних. Проаналізовано архітектурний стиль на основі Actor Model, описано основні компоненти системи та простежено шлях повідомлення від відправника до отримувача. Оцінено рішення щодо масштабованості. Зроблено висновок, що архітектура Discord є прикладом прагматичного інженерного підходу, де кожен компонент обирається під конкретну задачу, а система здатна розвиватись без зупинки сервісу.
